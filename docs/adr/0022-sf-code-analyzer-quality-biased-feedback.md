# ADR 0022: SF Code Analyzer uses quality-biased feedback

SF Code Analyzer should make every scan visible to the human without dumping verbose reports into the transcript, and should give the agent enough actionable violation context to repair code well. The default result shape is compact but expandable for humans, while LLM-facing feedback uses a quality-biased bounded detail profile: summary counts by severity and engine, all severity 1–2 findings, up to 10 severity 3 findings, and up to 5 severity 4–5 findings when they are in changed files and likely easy to fix. Each included finding carries file, line/column, engine/rule, severity, message, primary related location/comment, first resource URL, and first available fix/suggestion preview. Automatic and explicit scans default to session-scoped report artifacts outside the project tree unless an output path is requested; complete JSON/HTML/SARIF/CSV outputs are referenced by path and can be recovered from branch-local tool-result details. V1 requests and surfaces fixes or suggestions when available, but does not auto-apply them or add a separate mutating fix tool; agents use Pi's native file-editing tools for any chosen repair.
