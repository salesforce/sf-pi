# ADR 0021: SF Code Analyzer separates targeted auto-scans from explicit broad scans

SF Code Analyzer records supported files changed by agent `write` or `edit` tool results, then runs automatic Code Analyzer as a deferred post-agent quality pass instead of inline file-edit feedback. The deferred scan runs after Pi reports the agent has settled, so automatic retry, compaction retry, or queued follow-up work can finish before quality/security findings steer a repair pass. Project-wide, security, AppExchange, dependency, duplication, and other broad scans remain explicit command or tool actions.

Targeted auto-scans use a fast, file-suitable default profile (`pmd:Recommended` for Apex-like files, `eslint:Recommended` for JS/TS, `flow:Recommended` for Flow metadata, and `regex:Recommended` only where appropriate) rather than upstream `Recommended` across every engine. Changed files are grouped by selector profile and every group runs in parallel; each group gets its own transcript rows and report artifact, then findings are aggregated into one repair follow-up. Code Analyzer engines have different cost and scope profiles, and some engines need workspace context even when a target is supplied; keeping broad scans explicit preserves transparency while still giving agents timely feedback after they finish an edit pass. The auto-scan collector observes Pi tool results only, not arbitrary editor saves, and can be disabled through SF Pi-owned project or global preferences.

Clean deferred scan results produce compact human-visible feedback only. Actionable violations or setup failures produce compact human-visible feedback plus a bounded follow-up message that lets the LLM decide how to repair the findings. Automatic deferred targeted scans use a 30-second wall-clock soft runtime budget across local selector groups; if a group exceeds the remaining budget, SF Code Analyzer reports an ambiguous timeout result for that group and points the user or agent to an explicit scan for those targets. Automatic repair may continue without a fixed numeric cap, but only while there is demonstrable progress: a repair pass must edit relevant files and the next violation signature must change. The loop stops when Code Analyzer is clean, the agent makes no relevant edits, the violation signature repeats, setup/tool failures repeat, the user interrupts, or auto-scan is disabled; the stop reason is shown clearly. Explicit `code_analyzer` tool calls still run immediately because the user or agent asked for that scan at that point in the workflow.
